# ADR-002 – Folder Structure (Updated)

## Context
Define a scalable and modular folder structure for both backend and frontend applications.
This ADR replaces the previous structure to align with a feature-based architecture on the frontend, improving maintainability and scalability.

## Decision
**Backend:**
src/
├─ modules/
├─ config/
├─ middlewares/
├─ utils/
├─ app.ts
└─ server.ts

**Frontend:**
src/
├─ hooks/                # Global reusable hooks (e.g., useAuth, useFetch)
├─ modules/              # Feature-based modules
│   ├─ auth/
│   │   ├─ components/   # Auth-specific components (e.g., ProtectedRoute)
│   │   ├─ pages/        # Auth-related pages (e.g., LoginPage, RegisterPage)
│   │   └─ services/     # Auth logic (e.g., authServices.ts)
│   └─ otherFeature/
│       ├─ components/
│       ├─ pages/
│       └─ services/
├─ components/           # Global shared components
└─ App.tsx


## Justification
- Adopts a feature-based structure to improve scalability and separation of concerns.

- Keeps component-level styles co-located for easier maintenance and isolation.

- Centralizes shared hooks and components for reuse across features.

- Reduces coupling and improves discoverability of related logic.

## Consequences
- ✅ Easier scaling as new features are added.

- ✅ Clear separation between global and feature-specific logic.

- ✅ Better code readability and maintainability.

- ❌ Slightly more verbose directory tree, but improved long-term clarity.

# ADR-002 – Organização das Pastas (Atualizada) (PT)

## Contexto
Define uma estrutura de pastas escalável e modular para backend e frontend.
Esta ADR substitui a estrutura anterior, adotando uma arquitetura baseada em features no frontend.

## Decisão
**Backend:**
src/
├─ modules/
├─ config/
├─ middlewares/
├─ utils/
├─ app.ts
└─ server.ts

**Frontend:**
src/
├─ hooks/                # Hooks globais reutilizáveis (ex: useAuth, useFetch)
├─ modules/              # Módulos baseados em funcionalidades
│   ├─ auth/
│   │   ├─ components/   # Componentes específicos da feature
│   │   ├─ pages/        # Páginas relacionadas à autenticação
│   │   └─ services/     # Lógica e chamadas de API da feature
│   └─ outraFeature/
│       ├─ components/
│       ├─ pages/
│       └─ services/
├─ components/           # Componentes globais compartilhados
└─ App.tsx


## Justificativa
- Estrutura baseada em features aumenta a organização e a escalabilidade.

- Estilos por componente evitam conflitos e melhoram o isolamento visual.

- Hooks e componentes globais centralizados promovem reuso.

- Facilita a manutenção e o entendimento do código por novos desenvolvedores.

## Consequências
- ✅ Projeto mais modular e de fácil evolução.

- ✅ Separação clara entre lógica global e específica.

- ✅ Melhoria na legibilidade e manutenibilidade.

- ❌ Estrutura um pouco mais extensa, mas mais sustentável a longo prazo.
